# ADR-0001：可追溯文档结构方案

## 状态

已采纳

## 背景

项目希望满足以下要求：

- 每次更新都保留历史
- 每次更新都能说明更新了什么
- 每次更新都能说明还差什么、建议什么、风险是什么
- 后续 AI 可以快速检索到这些信息

一个直觉做法是每次更新复制一整份源码，形成多个版本目录。

## 决策

本项目不采用“每次复制整套源码目录”的方式保留历史，而采用以下组合方案：

1. 使用 Git Commit 保留代码级历史
2. 使用 Git Tag 标记版本节点
3. 使用 GitHub Release 保存安装产物
4. 使用 `docs/releases/` 保存结构化版本说明
5. 使用 `docs/rules/` 保存长期稳定规则
6. 使用 `docs/context/ai-retrieval-index.md` 作为 AI 检索入口

## 原因

选择该方案的原因如下：

- 不会让源码目录无限膨胀
- 更适合持续开发
- 更适合差异对比
- 更适合 AI 检索固定入口
- 更容易维护版本说明和风险说明

## 被放弃的方案

### 方案 A：每次复制一整份源码目录

放弃原因：

- 容易产生大量重复文件
- 难以维护
- 难以快速看出真实差异
- 容易让 AI 误判当前应该修改哪一份源码

### 方案 B：只保留 Git 历史，不写版本文档

放弃原因：

- 对人类和 AI 都不友好
- 无法快速回答“这次更新做了什么、风险是什么”
- 缺乏结构化记录

## 影响

采用本决策后，后续版本更新应默认同时维护：

- 代码改动
- 版本文档
- 风险说明
- 文件索引

## 风险

如果后续只改代码不更新文档，则该结构会失效。

因此需要把“更新文档”视为正式交付的一部分，而不是附加项。
