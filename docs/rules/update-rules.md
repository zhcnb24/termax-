# 更新规则

## 1. 目标

本规则用于确保项目每次更新都具备可追踪、可说明、可检索、可回滚的基本能力。

核心原则：

- 不只更新代码
- 还要更新记录
- 不只说明结果
- 还要说明过程和影响

## 2. 每次更新的最低要求

每次正式更新至少应补充以下信息：

- 更新了什么
- 为什么更新
- 改动了哪些文件
- 还差什么
- 建议什么
- 可能会发生什么
- 是否可回滚

## 3. 文档落点

每次版本更新建议在 `docs/releases/` 下至少维护以下文件：

- `vX.Y.Z.md`
  - 版本总说明
- `vX.Y.Z-checklist.md`
  - 完成项、未完成项、待验证项
- `vX.Y.Z-risk.md`
  - 风险、兼容性、潜在影响
- `vX.Y.Z-files.md`
  - 改动文件索引

如有必要，可额外补充：

- `vX.Y.Z-suggestion.md`
- `vX.Y.Z-rollback.md`
- `vX.Y.Z-compatibility.md`

## 4. 更新记录内容标准

### 4.1 更新了什么

明确列出新增、修改、删除、发布动作。

### 4.2 还差什么

明确列出未完成能力、已知缺口、需要后续推进的模块。

### 4.3 建议什么

明确列出下一步建议和推荐顺序。

### 4.4 可能会发生什么

明确列出潜在兼容性、性能、网络、权限、发布、数据层面的风险。

## 5. 文件保留原则

“保留上一次更新的文件”不等于复制整套源码目录。

推荐使用以下方式保留历史：

- 使用 Git Commit 保留代码历史
- 使用 Git Tag 标记版本
- 使用 GitHub Release 保存产物
- 使用 `docs/releases/` 保存结构化说明

## 6. 不推荐做法

以下做法原则上不推荐：

- 每次版本复制一整份源码目录
- 只发 APK 不写版本记录
- 只改代码不写说明
- 只说已完成，不说未完成和风险

## 7. AI 执行要求

AI 在完成较大变更后，应尽量同步补齐：

- 版本说明
- 改动文件说明
- 风险说明
- 未完成项说明

## 8. 样板顺序

建议未来每次版本文档按以下顺序准备：

1. 版本概览
2. 本次更新
3. 改动文件
4. 未完成项
5. 风险与影响
6. 下一步建议

## 9. 可回滚原则

如果更新影响以下内容，应明确说明回滚条件：

- 构建配置
- 签名方式
- 数据结构
- 包名或版本升级路径
- 权限申请逻辑
