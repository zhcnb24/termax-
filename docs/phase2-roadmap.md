# 第二阶段开发路线图

## 目标

把 `Termux千秋辅助` 从“原型展示”推进到“真实可用的最小版本”。

这一阶段不追求一次性做完所有工业级能力，只打通最关键的核心链路：

1. 用户提出目标
2. AI 给出回复或执行建议
3. 系统记录任务状态
4. 用户执行命令或文件操作
5. 系统展示输出、错误和日志
6. 本地持久化保存结果

---

## 本阶段范围

### 已实现

- `AI 聊天`
  - OpenAI 兼容接口配置
  - 离线回退回复
- `任务中心`
  - 任务状态流转
  - 日志记录
  - 失败状态与重试次数展示
- `环境检测`
  - Termux / Termux API / Shizuku
- `命令执行`
  - Shell 命令执行
  - stdout / stderr / exitCode 展示
- `文件系统能力`
  - 私有工作区创建 / 编辑 / 删除文件
- `本地存储`
  - SQLite 保存消息、任务、日志和配置

### 暂未实现

- 真正的 Agent 自动拆任务
- 自动判断风险并弹窗确认
- Termux API 实际调用
- Intent 执行通道
- Shizuku 授权与实际执行
- 后台任务恢复和持续通知
- 知识库导入
- 长期记忆分层

## 页面与职责

### 首页

- 展示第二阶段当前完成度

### AI聊天

- 展示命令输出和错误
- 创建、覆盖、删除工作区文件

### 任务中心

- 展示每次聊天、命令、文件操作对应的任务
- 展示最近日志
- 展示执行结果和失败信息

### 我的

- 展示当前存储能力
- 展示下一阶段规划

## 当前技术实现

### 数据层

- `SQLiteOpenHelper`
  - `messages`
  - `tasks`
  - `task_logs`
  - `settings`

### 运行层

- `EnvironmentDetector`
  - 检测安装包状态
- `ShellExecutor`
  - 基础命令执行
  - 20 秒超时
  - 基础自动重试
- `FileWorkspaceManager`
  - 私有工作区文件读写删
- `AiGateway`
  - OpenAI 兼容接口请求
  - 离线回退回复

### UI 层

- `QianQiuViewModel`
  - 统一管理消息、任务、日志、环境状态、工作区状态
- `QianQiuApp`
  - 5 个一级页面
  - 真实状态驱动

---

## 第三阶段建议切入顺序

1. `Agent Planner`
   - 根据用户目标自动生成步骤列表
2. `Guard`
   - 危险操作前确认
3. `Verifier`
   - 执行后检查结果是否达标
4. `Repairer`
   - 失败时根据错误自动修复并重试
5. `Mode Switch`
   - 普通模式 / 专家模式 / 开发者模式

---

## 结论

第二阶段的意义不是做“全能体”，而是让项目真正进入“能工作”的区间。

只要这一层稳定了，第三阶段开始叠加 Agent 能力时，整个系统才不会继续停留在漂亮但空心的原型状态。
